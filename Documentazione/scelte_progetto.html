Scelte di Progetto (Caso A)

Ristrutturazione del Progetto 1 con Python, Django e Bootstrap.

In due righe: abbiamo riscritto il progetto “Museo” usando Django (framework web in Python) e Bootstrap (stile front‑end) per avere codice più ordinato, pagine più accessibili e un’installazione locale più semplice (in 5 minuti).

1) Obiettivi

  • Funzionalità: mantenere tutte le pagine e il CRUD sulle Opere.
  • Qualità: interfaccia coerente, validazioni chiare, messaggi non invasivi.
  • Installazione: esecuzione da riga di comando o wizard PowerShell, accesso locale.

2) Architettura scelta

  • Django lato server: gestisce URL, logica e accesso al DB.
  • PostgreSQL per i dati: tabelle per Opere, Autori, Sale, Temi.
  • Template Django + Bootstrap 5 per le pagine.
  • File statici (CSS/JS) serviti via CDN e localmente con collectstatic + WhiteNoise.

3) Perché proprio queste tecnologie

  • Django ha un’architettura MTV (Model–Template–View): separa bene i pezzi dell’app, rende il codice leggibile e testabile.
  • ORM di Django evita SQL ripetitivo e riduce errori: si scrive Python, non query complesse.
  • Bootstrap 5 dà un layout responsive subito pronto, accessibile e uniforme.
  • PostgreSQL è affidabile e già previsto dalle linee guida locali; forniamo anche un dump per riprodurre i dati.

4) Glossario

Sigla / TermineSignificaEsempio pratico
CRUD Create, Read, Update, Delete: inserire, leggere, modificare e cancellare dati. Nella pagina Opere puoi aggiungere un’opera, cercarla, correggerla o eliminarla.
MTV Model–Template–View: struttura interna di Django. Model=tabella Opera; View=funzione Python che prepara i dati; Template=HTML che li mostra.
ORM Object‑Relational Mapping: scrivi Python invece di SQL per interrogare il DB. Opera.objects.filter(titolo__icontains="Mona") al posto di una SELECT manuale.
CDN Content Delivery Network: carica librerie (es. Bootstrap) da server veloci. Link a Bootstrap nel <head> senza scaricarlo nel repo.
.env File con variabili segrete (utente DB, password, ecc.), non incluso nel repo. Evita di pubblicare password; l’app le legge all’avvio.
collectstatic Comando Django che raccoglie i file statici in un’unica cartella. Utile per servire CSS/JS in modo ordinato.
WhiteNoise Piccolo server statici integrato: serve file statici compressi. Migliora le prestazioni senza configurazioni complesse.
Dump SQL Esportazione del database in un file .sql. Permette al prof di importare i nostri dati in un click.
Wizard Finestra guidata che chiede percorso progetto e credenziali. Automatizza venv, requirements, DB e avvio server.

5) Scelte sul database

  • PostgreSQL locale: richiesto dalle linee guida; evitiamo problemi di rete con DB remoti.
  • Vincoli (anni, tipi, referenze) coerenti con lo schema “Museo”.
  • Dump fornito: facilita verifica e riproducibilità.

6) Sicurezza e configurazione

  • Credenziali in .env, caricate con python-dotenv.
  • Utente DB applicativo diverso da postgres (principio del minimo privilegio).
  • Niente dati sensibili nel repository (password, host privati).
Nota per la valutazione: il progetto si avvia da riga di comando e si usa in locale, come richiesto nelle linee guida. Il manuale e gli script guidano l’installazione entro i 5 minuti.
